# result_state(기대결과) 재작성 계약

## 목표
구조화 TC 각 스텝의 `result_state`(기대결과)를, 원본 의도를 유지하되 **결론적 검증문**으로
재작성한다. 원본은 절차와 기대결과가 섞여 있고 장문 덩어리인 경우가 많다. 이를 그대로 옮기지 말고,
**그 스텝의 액션 이후 확인해야 할 결과**만 남긴 깔끔한 검증 문장으로 다듬는다.

## 입력 (스텝 하나)
```
{ new_tc_id, step_no, title, action, target, target_name, target_parameter,
  result_screen, result_screen_name, precondition_labels, tc_purpose, current_result_state }
```
- `tc_purpose` = 원본 기대결과 전체(맥락 파악용)
- `current_result_state` = 현재(원문 복사에 가까운) 기대결과 — 이걸 재작성한다

## 출력 (스텝 하나)
```json
{ "new_tc_id": "CHAT_NEW_0464", "step_no": 1,
  "result_state": "파일 다운로드 시 신뢰할 수 없는 파일 경고 얼럿이 노출되어야 함 — \"친구가 아닌 사용자가 보낸 파일은 안전하지 않을 수 있습니다. 다운로드하시겠습니까?\"" }
```

## 규칙
1. **결론적 표현으로 끝낼 것**: …됨 / …노출됨 / …되어야 함 / …않음 / …유지됨 등. 상태(결과)를 단정.
2. **절차/지시 제거**: 번호목록(1. 2.), 경로 화살표(`>`), "탭 시 / 선택 시 / 진입 후" 같은 절차 표현,
   담당자 표기, "*상세 케이스는 …TC에서 수행" 같은 참고 노트는 삭제. (그 스텝의 액션은 이미
   action/target 으로 구조화돼 있으므로 문장에 절차를 다시 쓰지 않는다.)
3. **실측 UI 문구는 원문 그대로 보존**: 따옴표로 묶인 얼럿/토스트/버튼/안내 문구는 글자 그대로 유지
   (테스트 실측 데이터). 임의로 바꾸거나 요약하지 말 것.
4. **OS 분기**: `[Android]`/`[iOS]`로 결과가 실제로 다르면 `Android: … / iOS: …` 형태로 정리.
   단순 태그면 제거.
5. **한 스텝에 검증이 여러 개면**: 스텝을 쪼개지 말고, 하나의 result_state 안에서 간결히 정리한다.
   3개 이상이면 짧은 `- ` 불릿 여러 줄로 나열 가능. 핵심 검증은 빠뜨리지 말 것.
6. **사전값 활용**: 화면/타깃은 자연스러운 한국어 명칭(target_name, result_screen_name)을 활용해
   가독성 있게. 단 문장이 어색해질 정도로 ID를 억지로 넣지 말 것.
7. **없는 검증을 지어내지 말 것.** 원본에 있는 결과만. 근거가 부족하면 원본 취지를 최대한 보존.
8. `current_result_state` 가 이미 "…화면이 정상 노출되어야 함" 류의 자동생성 문구면,
   target/화면 명칭으로 조금 더 구체화하되 억지 확장 금지.
9. 출력 `result_state` 는 **절대 빈 문자열이 될 수 없다**. 항상 의미 있는 검증 문장.

## 출력 형식
`{"results": [ {new_tc_id, step_no, result_state}, ... ]}` — 입력 배치의 모든 스텝을 순서대로.
