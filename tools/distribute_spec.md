# result_state 분배(distribute) 계약

## 배경(문제)
다단계 TC에서 **모든 스텝의 result_state 가 동일**하게 복제된 경우가 있다.
원인: 셋업/네비게이션 스텝(검증 문구가 없던 자동생성 스텝)에까지 최종 검증문이 복제됨.
→ "한 스텝 = 한 목적"에 어긋난다. 각 스텝이 **자기 액션의 결과**를 갖도록 재분배한다.

## 목표
TC 전체(모든 스텝)를 보고, 각 스텝의 result_state 를 그 스텝 액션에 맞는 결과로 재작성한다.
- **셋업/네비게이션/이동 스텝**(BACK, OPEN, 재입장, 백업·재설치 등): 그 스텝의 **중간 결과**만.
  예) "채팅 목록으로 이동됨", "챗로그 복원이 완료됨", "채팅방에 재입장됨"
- **검증(VERIFY 등) 스텝**: 원본 기대결과의 **실제 검증 내용**을 배치.
- 검증 내용이 여러 개고 각기 다른 스텝에 대응하면 **스텝별로 나눠** 배치(같은 문장 복제 금지).
- 원본 기대결과가 통째로 마지막 검증 스텝에만 해당하면, 앞 스텝엔 중간 결과, 마지막에 전체 검증문.

## 입력 (TC 하나)
```
{ new_tc_id, title, tc_purpose, steps:[ {step_no, action, action_parameter, target, target_name,
  start_screen, result_screen, result_screen_name, current_result_state}, ... ] }
```

## 출력 (TC 하나)
```json
{ "new_tc_id":"CHAT_NEW_0024",
  "steps":[
    {"step_no":1, "result_state":"챗로그 백업 후 톡 재설치 및 복원이 완료됨"},
    {"step_no":2, "result_state":"리액션 아이템은 복원되지 않으나 채팅방 말풍선은 정상 노출되고 기능 사용 가능함 - 개별 말풍선에 리액션 아이템 선택 시 해당 말풍선 데이터가 복원되고 신규 데이터가 적용됨 - 채팅방을 나갔다 재입장 시 전체 말풍선 리액션 아이템 데이터를 복원함(복원 실패 사유가 존재하면 재입장에도 복원 불가)"}
  ]
}
```

## 규칙
1. **스텝 간 동일 문장 복제 금지.** 각 스텝은 서로 다른 결과.
2. 셋업/이동 스텝은 그 스텝 액션(BACK/OPEN/재입장 등)과 원본 step 흐름에 근거한 **중간 결과**만. 억지 확장 금지.
3. 검증 스텝에 원본 tc_purpose 의 실검증 내용을 배치. 여러 검증이 여러 스텝에 대응하면 나눠 배치.
4. 따옴표로 묶인 실측 UI 문구(얼럿/토스트/버튼)는 **글자 그대로 보존**.
5. 결론적 표현(…됨/노출됨/되어야 함/않음)으로 종결. result_state 는 절대 빈 문자열 금지.
6. 없는 검증 지어내기 금지. 근거가 부족한 셋업 스텝은 "…화면이 정상 노출됨" 수준으로 보수적으로.
7. 입력 steps 의 **모든 step_no 를 출력에 포함**(개수·순서 일치).

## 출력 형식
`{"results":[ <TC 출력>, ... ]}` — 입력 배치의 모든 TC를 순서대로 포함.
